Before taking your Next.js application to production, there are some optimizations and patterns you should consider implementing for the best user experience, performance, and security.

This page provides best practices that you can use as a reference when [building your application](#during-development) and [before going to production](#before-going-to-production), as well as the [automatic Next.js optimizations](#automatic-optimizations) you should be aware of.

## Automatic optimizations

These Next.js optimizations are enabled by default and require no configuration:

<AppOnly>

- **[Server Components](../01-getting-started/05-server-and-client-components.mdx):** Next.js uses Server Components by default. Server Components run on the server, and don't require JavaScript to render on the client. As such, they have no impact on the size of your client-side JavaScript bundles. You can then use [Client Components](../01-getting-started/05-server-and-client-components.mdx) as needed for interactivity.
- **[Code-splitting](../01-getting-started/04-linking-and-navigating.mdx#how-navigation-works):** Server Components enable automatic code-splitting by route segments. You may also consider [lazy loading](lazy-loading.mdx) Client Components and third-party libraries, where appropriate.
- **[Prefetching](../01-getting-started/04-linking-and-navigating.mdx#prefetching):** When a link to a new route enters the user's viewport, Next.js prefetches the route in background. This makes navigation to new routes almost instant. You can opt out of prefetching, where appropriate.
- **[Prerendering](../04-glossary.mdx#prerendering):** Next.js prerenders Server and Client Components on the server at build time and caches the rendered result to improve your application's performance. You can opt into [Dynamic Rendering](../04-glossary.mdx#dynamic-rendering) for specific routes, where appropriate. {/* TODO: Update when PPR is stable */}
- **[Caching](../01-getting-started/08-caching.mdx):** Next.js caches data requests, the rendered result of Server and Client Components, static assets, and more, to reduce the number of network requests to your server, database, and backend services. You may opt out of caching, where appropriate.

</AppOnly>

<PagesOnly>

- **[Code-splitting](../../02-pages/03-building-your-application/01-routing/01-pages-and-layouts.mdx):** Next.js automatically code-splits your application code by pages. This means only the code needed for the current page is loaded on navigation. You may also consider [lazy loading](../../02-pages/02-guides/lazy-loading.mdx) third-party libraries, where appropriate.
- **[Prefetching](../../02-pages/04-api-reference/01-components/link.mdx#prefetch):** When a link to a new route enters the user's viewport, Next.js prefetches the route in background. This makes navigation to new routes almost instant. You can opt out of prefetching, where appropriate.
- **[Automatic Static Optimization](../../02-pages/03-building-your-application/02-rendering/04-automatic-static-optimization.mdx):** Next.js automatically determines that a page is static (can be prerendered) if it has no blocking data requirements. Optimized pages can be cached, and served to the end-user from multiple CDN locations. You may opt into [Server-side Rendering](../../02-pages/03-building-your-application/03-data-fetching/03-get-server-side-props.mdx), where appropriate.

</PagesOnly>

These defaults aim to improve your application's performance, and reduce the cost and amount of data transferred on each network request.

## During development

While building your application, we recommend using the following features to ensure the best performance and user experience:

### Routing and rendering

<AppOnly>

- **[Layouts](../03-api-reference/03-file-conventions/layout.mdx):** Use layouts to share UI across pages and enable [partial rendering](../01-getting-started/04-linking-and-navigating.mdx#client-side-transitions) on navigation.
- **[`<Link>` component](../03-api-reference/02-components/link.mdx):** Use the `<Link>` component for [client-side navigation and prefetching](../01-getting-started/04-linking-and-navigating.mdx#how-navigation-works).
- **[Error Handling](../01-getting-started/10-error-handling.mdx):** Gracefully handle [catch-all errors](../01-getting-started/10-error-handling.mdx) and [404 errors](../03-api-reference/03-file-conventions/not-found.mdx) in production by creating custom error pages.
- **[Client and Server Components](../01-getting-started/05-server-and-client-components.mdx#examples):** Follow the recommended composition patterns for Server and Client Components, and check the placement of your [`"use client"` boundaries](../01-getting-started/05-server-and-client-components.mdx#reducing-js-bundle-size) to avoid unnecessarily increasing your client-side JavaScript bundle.
- **Request-time APIs:** Be aware that Request-time APIs like [`cookies`](../03-api-reference/04-functions/cookies.mdx) and the [`searchParams`](../03-api-reference/03-file-conventions/page.mdx#searchparams-optional) prop will opt the entire route into [Dynamic Rendering](../04-glossary.mdx#dynamic-rendering) (or your whole application if used in the [Root Layout](../03-api-reference/03-file-conventions/layout.mdx#root-layout)). Ensure Request-time API usage is intentional and wrap them in `<Suspense>` boundaries where appropriate.

> **Good to know**: [Partial Prerendering (experimental)](/blog/next-14#partial-prerendering-preview) will allow parts of a route to be dynamic without opting the whole route into dynamic rendering.

</AppOnly>

<PagesOnly>

- **[`<Link>` component](../../02-pages/03-building-your-application/01-routing/03-linking-and-navigating.mdx):** Use the `<Link>` component for client-side navigation and prefetching.
- **[Custom Errors](../../02-pages/03-building-your-application/01-routing/08-custom-error.mdx):** Gracefully handle [500](../../02-pages/03-building-your-application/01-routing/08-custom-error.mdx#500-page) and [404 errors](../../02-pages/03-building-your-application/01-routing/08-custom-error.mdx#404-page)

</PagesOnly>

### Data fetching and caching

<AppOnly>

- **[Server Components](../01-getting-started/06-fetching-data.mdx):** Leverage the benefits of fetching data on the server using Server Components.
- **[Route Handlers](../03-api-reference/03-file-conventions/route.mdx):** Use Route Handlers to access your backend resources from Client Components. But do not call Route Handlers from Server Components to avoid an additional server request.
- **[Streaming](../03-api-reference/03-file-conventions/loading.mdx):** Use Loading UI and React Suspense to progressively send UI from the server to the client, and prevent the whole route from blocking while data is being fetched.
- **[Parallel Data Fetching](../01-getting-started/06-fetching-data.mdx#parallel-data-fetching):** Reduce network waterfalls by fetching data in parallel, where appropriate.
- **[Data Caching](../01-getting-started/08-caching.mdx):** Verify whether your data requests are being cached or not, and opt into caching, where appropriate. Ensure requests that don't use `fetch` are [cached](../03-api-reference/04-functions/unstable_cache.mdx).
- **[Static Images](../03-api-reference/03-file-conventions/public-folder.mdx):** Use the `public` directory to automatically cache your application's static assets, e.g. images.

</AppOnly>

<PagesOnly>

- **[API Routes](../../02-pages/03-building-your-application/01-routing/07-api-routes.mdx):** Use Route Handlers to access your backend resources, and prevent sensitive secrets from being exposed to the client.
- **[Data Caching](../../02-pages/03-building-your-application/03-data-fetching/01-get-static-props.mdx):** Verify whether your data requests are being cached or not, and opt into caching, where appropriate. Ensure requests that don't use `getStaticProps` are cached where appropriate.
- **[Incremental Static Regeneration](../../02-pages/02-guides/incremental-static-regeneration.mdx):** Use Incremental Static Regeneration to update static pages after they've been built, without rebuilding your entire site.
- **[Static Images](../../02-pages/04-api-reference/02-file-conventions/public-folder.mdx):** Use the `public` directory to automatically cache your application's static assets, e.g. images.

</PagesOnly>

### UI and accessibility

<AppOnly>

- **[Forms and Validation](forms.mdx):** Use Server Actions to handle form submissions, server-side validation, and handle errors.
- **[Global Error UI](../03-api-reference/03-file-conventions/error.mdx#global-error):** Add `app/global-error.tsx` to provide consistent, accessible fallback UI and recovery for uncaught errors across your app.
- **[Global 404](../03-api-reference/03-file-conventions/not-found.mdx#global-not-foundjs-experimental):** Add `app/global-not-found.tsx` to serve an accessible 404 for unmatched routes across your app.

</AppOnly>

- **[Font Module](../03-api-reference/02-components/font.mdx):** Optimize fonts by using the Font Module, which automatically hosts your font files with other static assets, removes external network requests, and reduces [layout shift](https://web.dev/articles/cls).
- **[`<Image>` Component](../03-api-reference/02-components/image.mdx):** Optimize images by using the Image Component, which automatically optimizes images, prevents layout shift, and serves them in modern formats like WebP.
- **[`<Script>` Component](scripts.mdx):** Optimize third-party scripts by using the Script Component, which automatically defers scripts and prevents them from blocking the main thread.
- **[ESLint](../../03-architecture/accessibility.mdx#linting):** Use the built-in `eslint-plugin-jsx-a11y` plugin to catch accessibility issues early.

### Security

<AppOnly>

- **[Tainting](../03-api-reference/05-config/01-next-config-js/taint.mdx):** Prevent sensitive data from being exposed to the client by tainting data objects and/or specific values.
- **[Server Actions](../01-getting-started/07-mutating-data.mdx):** Verify authentication and authorization inside each action. Do not rely on Proxy or layout or page level checks alone. Move database access to a `server-only` [Data Access Layer](data-security.mdx#data-access-layer) and consider [rate limiting](backend-for-frontend.mdx#rate-limiting) for expensive operations. Review the recommended [security practices](/blog/security-nextjs-server-components-actions).

</AppOnly>

- **[Environment Variables](environment-variables.mdx):** Ensure your `.env.*` files are added to `.gitignore` and only public variables are prefixed with `NEXT_PUBLIC_`.
- **[Content Security Policy](content-security-policy.mdx):** Consider adding a Content Security Policy to protect your application against various security threats such as cross-site scripting, clickjacking, and other code injection attacks.

### Metadata and SEO

<AppOnly>

- **[Metadata API](../01-getting-started/14-metadata-and-og-images.mdx):** Use the Metadata API to improve your application's Search Engine Optimization (SEO) by adding page titles, descriptions, and more.
- **[Open Graph (OG) images](../03-api-reference/03-file-conventions/01-metadata/opengraph-image.mdx):** Create OG images to prepare your application for social sharing.
- **[Sitemaps](../03-api-reference/04-functions/generate-sitemaps.mdx) and [Robots](../03-api-reference/03-file-conventions/01-metadata/robots.mdx):** Help Search Engines crawl and index your pages by generating sitemaps and robots files.

</AppOnly>

<PagesOnly>

- **[`<Head>` Component](../../02-pages/04-api-reference/01-components/head.mdx):** Use the `next/head` component to add page titles, descriptions, and more.

</PagesOnly>

### Type safety

- **TypeScript and [TS Plugin](../03-api-reference/05-config/02-typescript.mdx):** Use TypeScript and the TypeScript plugin for better type-safety, and to help you catch errors early.

## Before going to production

Before going to production, you can run `next build` to build your application locally and catch any build errors, then run `next start` to measure the performance of your application in a production-like environment.

### Core Web Vitals

- **[Lighthouse](https://developers.google.com/web/tools/lighthouse):** Run lighthouse in incognito to gain a better understanding of how your users will experience your site, and to identify areas for improvement. This is a simulated test and should be paired with looking at field data (such as Core Web Vitals).

<AppOnly>

- **[`useReportWebVitals` hook](../03-api-reference/04-functions/use-report-web-vitals.mdx):** Use this hook to send [Core Web Vitals](https://web.dev/articles/vitals) data to analytics tools.

</AppOnly>

### Analyzing bundles

Use the [`@next/bundle-analyzer` plugin](package-bundling.mdx#nextbundle-analyzer-for-webpack) to analyze the size of your JavaScript bundles and identify large modules and dependencies that might be impacting your application's performance.

Additionally, the following tools can help you understand the impact of adding new dependencies to your application:

- [Import Cost](https://marketplace.visualstudio.com/items?itemName=wix.vscode-import-cost)
- [Package Phobia](https://packagephobia.com/)
- [Bundle Phobia](https://bundlephobia.com/)
- [bundlejs](https://bundlejs.com/)
