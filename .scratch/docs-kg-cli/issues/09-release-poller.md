# 09 — Release poller (VPS) + Telegram ping (v1-optional, last)

**What to build:** The poller keeps the catalog fresh on its own so the operator never has to hand-watch upstream repos. Once a day on the VPS (its own box under ADR-0004, kept off Cloudflare because a missed run during the VPS's maintenance window is harmless), it walks every doc in the catalog and asks that doc's GitHub whether a newer release exists. It does this cheaply: one authenticated conditional GET per doc (`releases/latest`, falling back to `/tags` for tag-only repos), sending an ETag so an unchanged repo returns 304 and costs no rate budget, running serially/throttled so it never trips GitHub's secondary limits. When it sees a new tag it consults the version playbook (`version-resolution.jsonl`) to decide whether this is genuinely a new document — the tag moves the project's release axis and is not a pre-release (rc/canary/beta/`__branch__`) or a below-axis patch. A qualifying new version gets its license re-checked, is enqueued as a priority build for the build machine, and fires a Telegram ping to the operator (operator alerts are Telegram, never users' email). The whole job is resumable — it stores `{etag, last_seen_tag}` per doc plus a run cursor — so a run killed mid-way or during VPS maintenance resumes without re-polling finished docs or re-triggering seen tags. Launch-optional, lands last: the CLI, site, and backend all work without it; until it ships, new-version detection is manual.

**Blocked by:** 02 (walks the catalog store; enqueues via `/catalog/upsert` / the pipeline build-queue interface).

**Status:** ready-for-agent

- [ ] Runs as a once-daily scheduled job on the VPS (n8n/cronicle) iterating every doc in the catalog serially/throttled; a full dry-run pass over the current catalog completes with throttling params set within GitHub secondary-limit guidance.
- [ ] Each per-doc check is an AUTHENTICATED conditional GET of `releases/latest` (fallback `/tags`) sending the stored ETag; an unchanged repo returns 304 and consumes no rate-limit budget (verify: token on request, 304 response, rate-limit-remaining unchanged).
- [ ] Only a genuinely-new axis-level version triggers action: pre-release tags (rc/canary/beta/`__branch__`) and below-axis bumps are filtered out via `version-resolution.jsonl` (demo: a canary/patch tag produces no build; a real new axis-level release does).
- [ ] A qualifying new version re-checks that doc's license; on green it enqueues a PRIORITY build for the build machine (via the pipeline build-queue interface — the poller writes the enqueue signal, it does not run graphify itself) and sends a Telegram ping — and nothing to any user's email.
- [ ] If the license re-check returns **non-green**, no build is enqueued and that version is NOT shipped; the operator gets a Telegram alert flagging the license change (license-safety guard; pairs with 02's non-green `ready` block).
- [ ] State is persisted per doc as `{etag, last_seen_tag}` plus a run cursor, so a run killed mid-pass (or during VPS maintenance) resumes without re-polling checked docs or re-triggering an already-seen tag.
- [ ] The rest of the product works with the poller absent or disabled — no CLI/site/backend flow depends on it — confirming it is genuinely launch-optional.
